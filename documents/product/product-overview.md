## 비전

> Singo breaks down language barriers, so every Slack conversation is instantly understood in your preferred language.

**배경**: 회사는 다양한 언어(영어/일본어/한국어)를 사용하는데, 슬랙 채널에 여러 언어로 글이 올라와 읽기 힘들고 내용을 찾기 힘들다. 중요한 특정 슬랙 채널을 미러링해서 선호하는 언어로 번역하고, 놓치는 정보 없이 실시간 알림까지 보내주는 앱.

## 페르소나

*(각 페르소나를 아래 형식으로 추가. 페르소나 빌더 스킬로 작성.)*

### 신유진

> "글로벌 기업에서 일하며 일본 클라이언트와의 프로젝트를 맡고 있다. 한자가 어려워 일본어 슬랙 메시지를 놓치기 일쑤고, 영어도 비즈니스 레벨이라 읽고 쓰는 데 시간이 걸린다."

**Facts**
- 나이: 32세
- 직업: 글로벌 IT 기업 프로젝트 매니저 (일본 클라이언트 담당)
- 일본어 수준: 회화 가능(JLPT N2), 한자 읽기는 약함
- 영어 수준: 비즈니스 레벨이지만 원어민만큼 빠르게 읽지는 못함
- 소속 슬랙 워크스페이스: 본사(영어) + 일본 클라이언트 채널(일본어) 동시 소속

**Behaviors**
- 매일 아침 밤사이 쌓인 일본어 슬랙 메시지를 번역기로 하나씩 돌려본다
- 영어 채널의 빠른 논의를 따라가려 스크롤을 여러 번 되돌린다
- 슬랙 검색으로 예전 메시지를 다시 찾아보는 데 시간을 많이 쓴다
- 중요 채널은 즐겨찾기 해두고 수시로 들어가 새 글이 있는지 확인한다
- 회의 중에 일본어 메시지가 들어오면, 번역해보느라 회의의 흐름을 놓친다

**Needs & Goals**
- 일본어 메시지를 번역기 없이도 즉시 이해하고 싶다
- 여러 언어 채널을 오가며 맥락을 잃지 않고 일하고 싶다
- 중요한 메시지와 그냥 지나가는 대화를 구분해서 보고 싶다
- 번역·검색에 쓰는 시간을 줄이고 본업에 집중하고 싶다
- 팀에서 언어 때문에 소외되지 않는다는 소속감을 느끼고 싶다

★ **핵심 가정** *(고객 인터뷰로 검증 필요)*
- 신유진 같은 사람은 번역 품질보다 "속도(실시간성)"를 더 중요하게 여긴다
- 신유진은 특정 채널만 미러링해도 충분하다고 느낀다

## Problem 탐색

*(각 페르소나가 겪는 문제를 발산적으로 탐색한 결과)*

### 신유진

1. 일본어 메시지에 한자가 섞이면 무슨 내용인지 파악하는 데 시간이 오래 걸린다.
2. 회의 중 슬랙을 확인해도 내용 파악이 느려 회의 집중력이 떨어진다.
3. 여러 워크스페이스(본사 영어 채널 + 클라이언트 일본어 채널)를 오가며 맥락을 이어 붙이기 어렵다.
4. 어떤 메시지가 "지금 당장 봐야 하는 것"이고 어떤 게 "나중에 봐도 되는 것"인지 구분이 안 된다.
5. 번역기를 매번 수동으로 돌리는 게 번거롭고, 대화의 흐름(스레드 맥락)이 번역 과정에서 끊긴다.
6. 예전 대화를 다시 찾아보려 할 때, 원문 언어로 검색해야 해서 검색 자체가 어렵다.
7. 놓친 메시지에 뒤늦게 답하면서 클라이언트에게 신뢰를 잃을까 봐 불안하다.
8. 중요한 채널을 계속 열어놓고 지켜봐야 한다는 부담이 있다 (놓칠까 봐 상시 긴장).
9. 팀 내에서 "언어 때문에 정보 접근이 늦다"는 것 자체가 위축감으로 이어진다.

## Problem Priority

*(탐색된 문제 중 이 제품이 집중하기로 결정한 문제)*

이번 제품은 아래 두 가지에 집중한다 (신유진 Problem 탐색 1·4·5·8번 기반):

1. **실시간 자동 번역** — 한자·외국어 때문에 메시지를 즉시 이해하지 못하는 문제, 그리고 수동 번역으로 맥락이 끊기는 문제를 해결한다.
2. **선별적 실시간 알림** — 어떤 메시지가 중요한지 구분이 안 되고, 그래서 채널을 계속 지켜봐야 하는 부담을 해결한다.

(검색 어려움·신뢰 저하·위축감 등 나머지 문제는 위 두 가지 해결의 파생 효과로 본다.)

## 피처 탐색

*(밸류프로포지션을 실현하기 위한 기능 아이디어 전체)*

1. 지정한 슬랙 채널을 미러링해서 별도 뷰(또는 채널)에 원문+번역문을 나란히 보여준다.
2. 메시지가 올라오면 실시간으로 감지해 선호 언어로 자동 번역한다.
3. 번역된 메시지에 대해 즉시 알림(푸시/슬랙 DM)을 보낸다.
4. 알림 대상 채널/키워드를 사용자가 직접 지정할 수 있다 (선별적 알림).
5. 스레드 답글까지 맥락을 유지하며 번역한다 (스레드 구조 보존).
6. 번역된 내용을 원문 언어로도, 번역 언어로도 검색할 수 있다.
7. 여러 슬랙 워크스페이스(본사/클라이언트)를 하나의 통합 뷰에서 볼 수 있다.
8. 사용자별로 선호 언어를 설정할 수 있다 (다인 사용 시 팀원마다 다른 언어).
9. 번역 품질이 애매한 경우 원문을 함께 보여줘 확인할 수 있게 한다.
10. 중요도가 높다고 판단되는 메시지(멘션, 특정 키워드 등)에 우선순위 알림을 준다.
11. 놓친 메시지를 하루 요약본으로 받아볼 수 있다 (다이제스트).
12. 특정 채널만 선택적으로 미러링/구독할 수 있다 (전체가 아닌 일부).

## 기술 타당성 검토

| 피처 | 가능 여부 | 비고 |
|------|---------|------|
| 채널 선택적 미러링/구독 | ✅ 가능 | Slack Events API(봇 초대된 채널의 메시지 수신) + Supabase에 채널 설정 저장 |
| 실시간 자동 번역 | ✅ 확정 | **Claude API** — 한/영/일 업무 대화체·스레드 맥락 처리에 적합, LLM 기반이 아시아 언어 뉘앙스에 강함 |
| 원문+번역 나란히 보기 | ✅ 가능 | React 뷰 + Supabase에 원문·번역 함께 저장 |
| 즉시 알림 | ✅ 확정 | **Slack DM**으로 재전송 (별도 푸시/이메일 인프라 불필요, Slack Bot API로 구현) |
| 알림 대상 채널 지정 | ✅ 가능 | Supabase에 사용자별 설정 저장 |
| 스레드 맥락 유지 번역 | ✅ 가능 | Slack thread_ts로 스레드 재구성 가능 |

## Critical Paths

*(**e2e 발동 판단의 기준점** — `/story` 자가검증이 이 목록과 스토리를 대조해 e2e 여부를 정한다.)*

**정의**: Critical Path는 "유저 여정"이 아니라 **제품이 의존하는 끝-끝 통합 척추**다. 유저 여정(URL→화면)뿐 아니라 **데이터 척추**(webhook→ingest→DB 같은, 유저가 직접 안 밟지만 깨지면 치명적인 서버 통합)도 포함한다.

**슬롯 기준 — 통합 risk 필터**: 목록에 오르는 건 **핵심 risk가 통합 지점(realtime·transport·webhook·배포·브라우저 등)이라 유닛으로 못 잡는** 경로뿐이다. 순수 로직·격리 UI·단순 translate+렌더는 **유저 여정이라도 제외** — 유닛이 잡는다.

> ⚠️ **인플레이션 경계**: 이 목록이 기능 목록을 미러하기 시작하면 "모든 스토리 e2e"로 되돌아간다(우리가 탈출한 그 함정). **보수적으로 씨앗만**(보통 1개) 두고 시작하고, 회귀가 **실제로 발생해 증명될 때만** 키운다.

형식: `**[경로 이름]**: [끝-끝 흐름 한 줄] — [e2e 파일 경로 또는 "아직 없음"]`

- **메시지 도달**: Slack 메시지 발생 → 실시간 수신(Events API) → Claude API 번역 → 화면 렌더 — 아직 없음 *(realtime·외부 API 통합 risk — 유닛으로 못 잡음)*
